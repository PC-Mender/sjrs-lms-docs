## Overview

The SJRS LMS uses **Zod** as its sole validation library for all backend input validation. The previous custom validation engine (`StreamlinedValidationEngine`, `ValidationRule`, `ValidationConfig`) has been fully retired. All schemas are now Zod schemas, and validation runs through `schema.safeParse()`.

The system follows a **two-phase architecture**:
1. **Validation** — Zod schemas validate request body shape, types, and constraints
2. **Sanitization** — Post-validation XSS/SQL sanitization applied in security middleware

## Architecture

### **Backend Validation Pipeline**

```
Request → Security Middleware → Validation Handler → API Handler
                                   │
                                   ├─ 1. Parse JSON body
                                   ├─ 2. schema.safeParse(body)
                                   ├─ 3. If invalid → 400 error with field-level details
                                   ├─ 4. If valid → XSS/SQL sanitization (post-validation)
                                   └─ 5. Pass sanitizedData to handler
```

### **Key Files**

```
functions/middleware/validation/
├── core/
│   └── validation-engine.ts    # validateInput() — Zod safeParse wrapper
├── schemas/                     # 23 Zod schema files (one per resource)
│   ├── user-schemas.ts
│   ├── book-schemas.ts
│   ├── role-schemas.ts
│   ├── loan-schemas.ts
│   └── ... (20 more)
├── patterns/                    # Shared regex patterns (VALIDATION_PATTERNS)
├── sanitizers/                  # XSS/SQL sanitization utilities
├── types.ts                     # ValidationError, ValidationResult
└── index.ts                     # Barrel exports

functions/middleware/security/
└── handlers/validation-handler.ts  # Orchestrates validate + sanitize
```

### **Validation Engine**

The engine (`validation-engine.ts`) is a thin wrapper around `schema.safeParse()`:

```typescript
export async function validateInput(
  data: Record<string, unknown>,
  schema: ZodType,
  _config: Record<string, unknown> = {},
  env?: Environment
): Promise<ValidationResult> {
  const result = schema.safeParse(data);

  if (result.success) {
    return { isValid: true, errors: [], sanitizedData: result.data };
  } else {
    const zodErrors = extractZodIssues(result.error).map(err => ({
      field: Array.isArray(err.path) ? err.path.map(String).join('.') : '',
      message: String(err.message),
      code: err.code ? String(err.code).toUpperCase() : 'VALIDATION_ERROR',
    }));
    return { isValid: false, errors: zodErrors };
  }
}
```

The engine supports both Zod v3 (`.errors`) and Zod v4 (`.issues`) error formats via `extractZodIssues()`.

### **Validation Types**

```typescript
interface ValidationError {
  field: string;
  message: string;
  code: string;
}

interface ValidationResult {
  isValid: boolean;
  errors: ValidationError[];
  sanitizedData?: any;
}
```

## Writing Zod Schemas

### **Schema Conventions**

All backend schemas follow these patterns:

- Use `.passthrough()` on object schemas to match the system's unknown-field tolerance (extra fields are not rejected)
- Use `.optional()` for non-required fields
- Use `VALIDATION_PATTERNS` from `patterns/` for shared regex (email, phone, ISBN, year, password, etc.)
- Define create and update schemas separately — update schemas make all fields `.optional()`

### **Example: Creating a Schema**

```typescript
import { z } from 'zod';
import { VALIDATION_PATTERNS } from '../patterns';

export const USER_REGISTRATION_SCHEMA = z.object({
  email: z.string().email('Invalid email format').max(255),
  password: z.string().min(8, 'Password must be at least 8 characters')
    .regex(VALIDATION_PATTERNS.PASSWORD, 'Password must contain lowercase, uppercase, number, and special character'),
  first_name: z.string().min(2).max(50).regex(VALIDATION_PATTERNS.ALPHABETIC),
  last_name: z.string().min(2).max(50).regex(VALIDATION_PATTERNS.ALPHABETIC),
  user_type: z.string(),
  phone: z.string().regex(VALIDATION_PATTERNS.PHONE).optional(),
  turnstile_token: z.string().optional(),
}).passthrough();
```

### **Example: Update Schema (all fields optional)**

```typescript
export const USER_UPDATE_SCHEMA = z.object({
  first_name: z.string().min(2).max(50).regex(VALIDATION_PATTERNS.ALPHABETIC).optional(),
  last_name: z.string().min(2).max(50).regex(VALIDATION_PATTERNS.ALPHABETIC).optional(),
  email: z.string().email().max(255).optional(),
  phone: z.string().regex(VALIDATION_PATTERNS.PHONE).optional(),
  status: z.string().refine(
    (v) => ['active', 'inactive', 'suspended', 'pending'].includes(String(v)),
    'Invalid status value'
  ).optional(),
}).passthrough();
```

### **Available Validation Patterns**

Import from `functions/middleware/validation/patterns`:

| Pattern | Description |
|---------|-------------|
| `VALIDATION_PATTERNS.EMAIL` | Standard email format |
| `VALIDATION_PATTERNS.PASSWORD` | Lowercase + uppercase + number + special char |
| `VALIDATION_PATTERNS.PHONE` | International phone format |
| `VALIDATION_PATTERNS.ISBN` | ISBN-10 or ISBN-13 |
| `VALIDATION_PATTERNS.YEAR` | 4-digit year (1800–current+1) |
| `VALIDATION_PATTERNS.ALPHABETIC` | Letters, spaces, hyphens only |
| `VALIDATION_PATTERNS.ALPHANUMERIC` | Letters, numbers, hyphens, underscores |

## Using Schemas in Handlers

### **Wiring a Schema to an Endpoint**

Schemas are attached to endpoints via the `SecurityConfig` object in the security middleware:

```typescript
import { VALIDATION_SCHEMAS } from '../middleware/validation';
import { withSecurity } from '../middleware/security';

export async function handleCreateUser(request: Request, env: Environment) {
  return withSecurity(request, env, {
    validationSchema: VALIDATION_SCHEMAS.USER_CREATE,
    enableValidation: true,
    enableXSSProtection: true,
  }, async (req, env, { sanitizedData }) => {
    // sanitizedData is already validated + sanitized
    // Use sanitizedData directly — do NOT re-read request body
    const result = await createUser(sanitizedData, env);
    return createCreatedResponse(result, 'User created');
  });
}
```

### **Critical Rules**

- **Never re-read `request.json()` in the handler** — the validation handler consumes the body once and passes `sanitizedData` via the security context
- **Always use `sanitizedData`** from the security context, not the raw request body
- GET/DELETE/HEAD/OPTIONS requests skip body validation (no body to validate)

## Sanitization (Post-Validation)

Sanitization is applied **after** Zod validation succeeds, inside `validation-handler.ts`. This separation ensures schemas stay focused on shape/type validation.

### **How It Works**

1. Zod validation passes → `sanitizedData` contains validated data
2. For each field in `sanitizedData`:
   - **Password fields** (any key containing `password`) → **skipped** (must match exactly for auth; SQL injection prevented by D1 parameterized queries)
   - **Turnstile tokens** (any key containing `turnstile`) → **skipped** (must be submitted verbatim to Cloudflare)
   - **URL fields** (`url` or `*_url`) → `sanitizeURLInput()` (avoids entity-encoding slashes)
   - **All other fields** → `sanitizeXSSInput()` (HTML entity encoding)
3. Result is the final `sanitizedData` passed to the handler

### **Why Not Sanitize in Schemas?**

- Schemas define **what valid data looks like** (types, constraints)
- Sanitizers define **how to make data safe** (XSS encoding, SQL escaping)
- Keeping them separate means schema changes don't accidentally affect sanitization and vice versa
- Passwords and Turnstile tokens need to bypass sanitization — this is easier to manage as a post-validation step

## Frontend Response Validation

The frontend `d1Client.ts` optionally validates API responses using Zod schemas via `safeParse()`:

```typescript
const { data } = await d1Request('/api/v1/users', {
  schema: UserResponseSchema, // optional Zod schema
});

// In d1Client.ts:
if (schema) {
  const result = schema.safeParse(dataToValidate);
  if (!result.success) {
    console.warn(`[d1Request] Schema validation warning for ${path}: ${issues}`);
    // Returns original data on failure — non-breaking
  } else {
    validated = result.data;
  }
}
```

Response validation is **non-breaking**: schema mismatches produce `console.warn`, not throws. The original data is returned on failure so the UI continues to function.

## Adding a New Schema

1. **Create or edit** the appropriate file in `functions/middleware/validation/schemas/`
2. **Define the Zod schema** following the conventions above (`.passthrough()`, `.optional()` for non-required)
3. **Export from** `schemas/index.ts`
4. **Add to** `VALIDATION_SCHEMAS` in `common-schemas.ts` if it should be part of the aggregate
5. **Wire to handler** via `withSecurity({ validationSchema: ... })`

## Migration History

The system was migrated from a custom validation engine to Zod in four phases:

1. **Phase 1**: Converted backend schema files to Zod with `.passthrough()` and `.optional()` patterns
2. **Phase 2**: Enabled Zod response validation in `d1Client.ts` (non-breaking, `console.warn` on mismatch)
3. **Phase 3**: All 23 schema files in `functions/middleware/validation/schemas/` converted to Zod; sanitizers removed from schemas
4. **Phase 4**: ~30 inline custom schemas converted to Zod; custom validation engine retired; `ValidationSchema`, `ValidationRule`, `ValidationConfig` type exports removed

The custom `StreamlinedValidationEngine`, cache manager, and rule-based validation system no longer exist. All validation goes through `schema.safeParse()`.
