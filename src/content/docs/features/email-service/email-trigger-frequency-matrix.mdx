This page documents email trigger behavior based on backend code paths.

Scope notes:
- This is code-based behavior, not runtime analytics counts.
- Frequency values are general expectations per event path.
- Most transactional sends are also gated by user notification preferences (`email_notifications` and type-specific flags).

## Matrix

| Template | Trigger (When) | General Frequency (How Many Times) |
|---|---|---|
| `email-confirmation` | New registration, resend confirmation, email-change confirmation | Usually 1 per trigger attempt; resend paths are rate-limited |
| `password-reset` | Password reset request for an existing account | 1 per accepted reset request |
| `login-alert` | Successful login when IP or device or location changed versus previous session | 0 or 1 per login |
| `account-recovery-alert` | Account recovery requested, lockout event, or recovery completed | Up to 1 per event occurrence |
| `status-check-otp` | Public status-check OTP send endpoint | 1 per eligible send request |
| `mfa-email-otp` | MFA challenge email send and MFA email enrollment send | Usually 1 per request; cooldown may skip new send |
| `mfa-disable-otp` | MFA disable flow after password verification | Usually 1 per request; cooldown may skip new send |
| `mfa-replace-authenticator-otp` | Replace authenticator app flow after password verification | Usually 1 per request; cooldown may skip new send |
| `webauthn-revoke-otp` | Revoke WebAuthn credentials flow after password verification | Usually 1 per request; cooldown may skip new send |
| `borrow-notification` | Borrow/order approved path that creates borrower-facing approval email | 1 per approved order event per recipient |
| `loan-renewal-notification` | Renewal decision and loan renewed notifications | Usually 1 per renewal decision or renewal event |
| `return-reminder` | Scheduled due-date reminder worker (due in 3 days, 1 day, or today) | Up to 3 per active loan lifecycle (one at each checkpoint) |
| `overdue-notification` | Scheduled overdue reminder worker (7, 14, 30 days overdue) | Up to 3 per overdue loan lifecycle (one at each checkpoint) |
| `payment-success` | Payment receipt notification publish flow | 1 per successful payment event |
| `admin-approval` | Account status transition to active or inactive in admin user update flow | 1 per qualifying status transition |
| `profile-completion` | Successful profile completion/update submission paths | 1 per successful profile update request |
| `order-notification` | Order notification endpoint usage and order-created admin notification path | 1 per recipient per trigger (can fan out to multiple admins) |
| `system-notification` | Generic transactional/system events (order rejected, reservation events, penalty notices, loan status notices, reservation expiry warnings, bulk announcements) | Usually 1 per recipient per event; can be many in loops/batches |
